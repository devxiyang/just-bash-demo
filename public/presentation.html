让模型完成一次 P1 舆情应急处置，但别把机器交给它

这个 case 展示的是 Feature Agent：模型进入一个受控工程工作区，读取多源输入、实现归一化与升级决策代码、跑测试、生成决策 JSON 和管理层简报。just-bash 在这里不只是沙箱，而是文件系统、命令扩展、AST instrumentation 和安全边界组合出的 agent 运行时。

核心命题：代码生成不是终点，可验证交付才是终点
Build Feature

实现 `src/normalize.ts`、`src/risk-score.ts`、`src/escalation.ts`：从信号归一化走到 P1 判断。

Govern Agent

用测试、lint、guard、artifact 和沙箱边界约束模型行动。

05 / just-bash Stackjust-bash README
这页是补强重点：不要只讲 sandbox。just-bash 的亮点是把 shell、文件系统、命令扩展和 AST instrumentation 放在同一个 TypeScript 运行时里。
Filesystem Strategy

OverlayFS：读真实仓库，写入留在内存

copy-on-write workspace
const overlay = new OverlayFs({ root: "/path/to/project" });
const bash = new Bash({ fs: overlay, cwd: overlay.getMountPoint() });

await bash.exec("cat package.json");       // 从真实磁盘读
await bash.exec("echo changed > file.ts"); // 写入 overlay，留在内存

// 人类确认后，再决定是否把 diff 落盘
适合代码功能场景

agent 能像在真实 repo 里工作：读源码、跑测试、生成 patch。

默认不污染宿主机

读来自真实项目，写停在 overlay。可以先 review，再决定是否 apply。

可组合挂载

MountableFs 可以把只读知识库、临时工作区、真实数据目录组合成统一路径空间。

06 / OverlayFS当前 demo 为了稳定用 InMemoryFs；生产型演示可切到 OverlayFs
强调我们的 demo 用 InMemoryFS 是为了可重复，但真实“写代码做功能”非常适合 OverlayFS：读真实仓库，写入内存，人工确认后再落盘。
Custom Commands

自定义命令：把团队能力做成模型可组合的 shell 动词

defineCommand
const incidentAnalyze = defineCommand("incident-analyze", async (args, ctx) => {
  const data = await ctx.fs.readFile(args[0]);
  return { stdout: analyze(data), stderr: "", exitCode: 0 };
});

const incidentGuard = defineCommand("incident-guard", async (args, ctx) => {
  const report = await ctx.fs.readFile(args[0]);
  return checkPolicy(report);
});

不要只给模型“万能 shell”。更好的方式是把业务动作做成 可管控、可测试、可组合 的命令。

AST Instrumentation

AST Transform：在执行前理解和改写 bash

transform pipeline
const pipeline = new BashTransformPipeline()
  .use(new TeePlugin({ outputDir: "/tmp/logs" }))
  .use(new CommandCollectorPlugin());

const result = pipeline.transform("grep -R TODO src | head");

result.script;             // 改写后的 bash
result.metadata.commands;  // ["grep", "head"]
执行前审计

从 AST 提取命令列表，先判断会不会触碰网络、危险命令或大范围文件。

自动插桩

用 TeePlugin 捕获每段命令输出，把 shell 过程变成可回放证据。

元数据反馈

让 harness 不只看最终 stdout，也能看“模型用了哪些动作”。

Agent Loop

从需求到交付：一个 Feature Agent 的闭环

1. 观察读 README、需求、架构、分析策略、样本数据。
2. 失败运行测试，看到分类和聚合逻辑尚未实现。
3. 修改编辑 `src/normalize.ts`、`src/risk-score.ts`、`src/escalation.ts`。
4. 验证重新跑 `npm test`、`npm run lint`、`incident-guard`。
5. 交付生成 `reports/incident-brief.md` 和 `reports/escalation-decision.json`。

Production Checklist

如果把这个模式放进真实舆情系统

数据入口

把抓取、脱敏、去重、采样做成独立受控 pipeline。

分析核心

用测试集、golden cases、评估指标约束分类和风险提取。

报告治理

报告必须引用证据 id，避免逐字传播注入文本和敏感信息。

权限分层

抓取、分析、写报告、外发通知应是不同工具和不同审批路径。

人工升级

高风险舆情、证据冲突、guard 失败时默认交给人。

审计轨迹

保存 tool call、输出、artifact diff 和最终报告。

Takeaway

这不是给模型一个 shell，而是给模型一个受控工程工作台

舆情应急处置 case 的价值在于：它把 写代码、处理多源数据、依据规则做决策、生成 artifact、验证输出、安全边界 全部放进同一个可观察回路。

Observe

需求、架构、数据、日志。

Act

编辑代码、运行命令、生成报告。

Verify

测试、lint、guard、证据 id。

Steer

人定义标准，agent 在边界内执行。
